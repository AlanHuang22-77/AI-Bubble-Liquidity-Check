# 第14節待決問題選項與建議報告

> 本報告針對《月度流動性儀表板設計圖.md》§14 的 7 個待決問題，逐題整理可選方案與建議優先評估方向。本文不是最終決議，不修改原設計圖、不移除 §14 問題、不把文件升為正式 spec，也不啟動 Phase 1 實作。

## 0. 使用方式與建議決策順序

本報告中的「建議」應視為下一輪討論的工作假設，不是拍板答案。正式決議仍需由使用者確認後，才寫回《月度流動性儀表板設計圖.md》的對應章節，並依 §14 規則移入「已決議」。

建議先後順序：

1. 先確認資料源授權，因為它決定 EPS、0DTE、槓桿 ETF、終端機匯出資料能否合法入庫。
2. 再確認更新責任，因為它決定人工判讀、資料查核與月報審核是否能分工。
3. 接著確認實作載體，因為它會影響抓取模式、排程載體與匯出格式。
4. 最後確認情境機率呈現，因為這是研究口徑與產品文案問題，不應反向牽動資料模型。

## 1. 實作載體

待決問題：Web app、Streamlit、Google Sheets/Looker Studio，或 Obsidian/Markdown 儀表板？

判斷準則：

- 是否能清楚呈現 `data_as_of`、fresh/stale/missing、門檻、公式與 analyst_note。
- 是否能支援月度重算、缺值標示、Markdown 月報輸出。
- 是否適合目前 repo 的「研究文件與規格階段」，避免過早進入重型前端工程。
- 是否方便保留 CSV/JSON 作為可稽核資料層。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| Web app | 已確定多人使用、需要正式 UI、權限控管、可部署產品 | 可完整實作 §3、§4、§10 的互動規格；可建立元件與測試；長期產品化能力最強 | 前期成本最高；在 §14 未決前容易過早綁定框架；資料管線尚未穩定時，前端可能反覆重工 |
| Streamlit | 本地 MVP、研究者每月更新、需要表格與簡易互動 | 以 Python 快速連接 CSV/JSON、計分邏輯、表格、圖表與 Markdown 匯出；可先驗證指標與情境規則 | UI 細緻度與多人權限較弱；若日後要正式產品化，可能需要重寫成 Web app |
| Google Sheets/Looker Studio | 團隊習慣試算表、資料多半手動、需要快速分享 | 上手最快；人工輸入、審核與圖表很直覺；適合初期驗證門檻與月報流程 | 規則版本控管、測試、缺值邏輯與情境引擎較容易失控；長期可稽核性弱於程式化管線 |
| Obsidian/Markdown 儀表板 | 以研究筆記為主、低互動、重視可讀性與檔案保存 | 最貼近目前 repo；不需部署；月報與註記可直接沉澱為 Markdown | 不適合完整的分數重算、資料品質檢查、互動表格與情境矩陣 |

建議優先評估：`Streamlit 本地 MVP + Markdown/CSV/JSON 作為權威資料與報告層`。

理由：設計圖 §12 已把 Phase 1 定為本地 MVP、CSV/JSON 手動輸入、9 個核心指標與 Markdown 匯出。Streamlit 能用最低工程成本驗證「計分、stale/missing、情境引擎、月報匯出」是否成立，同時不犧牲日後轉 Web app 的可能性。

保留替代路徑：若主要使用者不想執行本地 Python，Google Sheets/Looker Studio 可以作為更低門檻的第一版；若目標已明確是多人部署產品，則直接選 Web app，但應先完成 §14 其餘決策。

決議前需確認：

- 儀表板主要是單人研究工具，還是需要多人同時使用？
- 使用者是否接受本機 Python/Streamlit 執行環境？
- 是否需要登入權限、雲端部署、分享連結或審核流程？
- 第一版是否以「驗證研究框架」優先，而非正式產品體驗優先？

## 2. 資料源授權

待決問題：是否有 Bloomberg、FactSet、Refinitiv，或必須完全使用公開資料與手動 CSV？

判斷準則：

- 官方與可稽核來源優先，符合《資料抓取與查證工作流.md》§1 的來源分級。
- 付費資料若進入 repo，需確認授權允許儲存、再加工、截圖或月報引用。
- 沒有授權時，不得用推估值冒充實際值；應標 `missing` 或使用明確註記的替代來源。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| 全付費資料源優先 | 有 Bloomberg、FactSet、Refinitiv 且授權允許研究入庫 | EPS、forward estimate、信用、ETF flow 等資料完整度較高；人工查核效率較好 | 授權限制可能禁止再散布；自動化 API 權限可能另收費；repo 內資料保存需更嚴格 |
| 完全公開資料與手動 CSV | 無終端機或付費授權，或只做個人研究 MVP | 最容易啟動；與 §12 Phase 1 manual_csv 相容；降低法律與部署阻塞 | NDX forward EPS、0DTE、槓桿 ETF、DAT mNAV 可能需要二手或人工來源；資料品質不均 |
| 混合模式 | 有部分付費資料或可人工摘錄，但大多數市場資料可公開取得 | 對官方公開資料自動化，對 EPS/質性資料保留人工欄位；兼顧可行性與精度 | 需要清楚標記 `source_tier`、`fetch_mode`、授權限制；月報引用需要更嚴格審核 |

建議優先評估：`混合模式`，但 Phase 1 應能在「公開資料 + 手動 CSV」條件下運作。

理由：核心指標中 HY/IG OAS、BTC、SOFR-IORB、ON RRP、TGA 等可使用公開或官方市場資料；但 NDX forward EPS、0DTE、槓桿 ETF、部分 AI 債務化資料，天然會遇到付費或延遲來源。混合模式能讓資料品質逐步升級，不把 MVP 卡死在單一資料授權答案上。

決議前需確認：

- 是否實際擁有 Bloomberg、FactSet、Refinitiv 或其他資料商授權？
- 授權是否允許把匯出值存入本地 CSV/JSON 與 Markdown 月報？
- 月報是否只供個人使用，還是會分享給第三方？
- 若付費資料不可入庫，是否接受該指標以 `manual_csv` 或 `stale/missing` 方式處理？

## 3. 資料抓取模式

待決問題：自動 API、手動 CSV，或混合？（Phase 1 預設以 manual_csv 啟動）

判斷準則：

- 自動化不能犧牲來源可稽核性。
- 質性判讀指標，如 FOMC 語調、EPS 指引方向，不應硬做成無註記的全自動判定。
- `stale` 與 `missing` 必須保留，不可被自動流程轉成 `neutral`。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| 全自動 API | 資料源穩定、欄位固定、無授權障礙 | 月度更新效率最高；可重跑、可測試、可留 log | 來源欄位變動會造成靜默錯誤；質性指標不適合全自動；初期建置成本較高 |
| 全手動 CSV | 先驗證研究框架、資料欄位仍在調整 | 最符合 §12 Phase 1；人可以先處理授權與來源口徑差異 | 容易輸入錯誤；長期維護成本高；若無檢查腳本，資料品質不穩 |
| 混合抓取 | 部分官方資料可自動，部分延遲/付費/質性資料需人工 | 最符合 SOP §2 的自動、半自動、人工分類；可逐步擴充 | 需要明確標記每個欄位由誰、何時、如何取得；流程管理較複雜 |

建議優先評估：`混合抓取，Phase 1 以 manual_csv 作為 canonical input`。

具體方向：第一版所有指標都能用 CSV/JSON 手動輸入；同時把 HY/IG OAS、BTC、SOFR-IORB、ON RRP、TGA 這類公開官方資料列為第一批自動化候選。FINRA、FOMC、NDX forward EPS、0DTE、槓桿 ETF、DAT/mNAV 先保留人工或半自動。

決議前需確認：

- 是否要把 `source_tier` 與 `fetch_mode` 加入 `indicator_definitions`？
- 是否需要每月產生抓取 log 與查核清單？
- 自動抓取失敗時，是停止月報、沿用上月值並標 `stale`，還是允許人工補件？
- CSV 是否需建立欄位驗證，例如 month、indicator_id、data_as_of、source_name 不可空白？

## 4. 自動抓取腳本執行載體

待決問題：本機 cron、GitHub Actions，或手動執行？（依實作載體決議後定，詳見工作流 §6）

判斷準則：

- 執行載體必須能保留更新時間、來源、錯誤與是否沿用上月值。
- 若使用付費資料或本機終端機匯出，不宜先假設可放到雲端 CI。
- 在實作載體未決前，不應過早決定排程系統。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| 手動執行 | Phase 1、資料源未穩、仍需人工查核 | 可控、容易除錯；符合月度節奏；不需處理雲端 secrets | 容易忘記執行；需要清楚的月度 checklist |
| 本機 cron | 單人研究環境、資料在本機、含付費終端機匯出 | 可自動提醒或跑公開資料；避免把私有資料放到第三方 | 本機睡眠、路徑、環境變數與失敗通知需處理；可觀測性弱 |
| GitHub Actions | 公開 API、無敏感授權、希望留 CI log | 執行紀錄清楚；易每月排程；可搭配 pull request 審核資料更新 | secrets 與授權限制需審查；付費資料不一定允許；抓取外站可能有速率或可用性問題 |
| App server scheduler | 已選 Web app 並部署後 | 可與 UI、資料庫、通知整合 | 對 MVP 過重；需處理部署、權限、監控與成本 |

建議優先評估：`Phase 1 手動執行，保留 log；穩定後依資料授權選 GitHub Actions 或本機 cron`。

理由：本專案目前是 Markdown 研究規格倉庫，沒有應用程式或測試管線。手動執行最能避免在資料來源、授權、平台都未定時引入排程複雜度。等第一批公開資料抓取穩定後，若資料全為公開來源，可再轉 GitHub Actions；若含本機終端機或私有檔案，則優先本機 cron。

決議前需確認：

- 自動化是否會碰到付費資料、cookie、終端機匯出或本機檔案？
- 是否需要失敗通知，例如 email、Slack、GitHub issue？
- 每月第 1 個工作日與 FINRA 發布後是否要分成兩個執行流程？
- 產出的 CSV/JSON 是直接覆寫，還是按月份保留版本？

## 5. 月報匯出格式

待決問題：Markdown、PDF、HTML，或多格式並行？（§11 已定 Markdown 為最低要求）

判斷準則：

- Markdown 已是 §11 最低要求，且最符合目前 repo 的文件型工作流。
- 衍生格式不可成為唯一權威版本，避免 PDF/HTML 與 Markdown 內容不一致。
- 匯出格式應支援資料品質揭露、四問摘要、觸發訊號與情境更新。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| Markdown only | 研究筆記、repo 保存、版本控管 | 最可稽核；容易 diff；與 §11 完全一致 | 對外分享的閱讀體驗較普通；圖表呈現有限 |
| PDF | 會議分享、歸檔、不可編輯版本 | 閱讀與分享友善；格式固定 | 生成流程需維護；若直接編 PDF，會破壞可追溯性 |
| HTML | Web app 或互動報告 | 可加入互動圖表與 drilldown | 若沒有 Web app，維護成本偏高；列印與歸檔需另處理 |
| 多格式並行 | 同時需要 repo 保存與對外分享 | Markdown 做權威源，PDF/HTML 做派生輸出 | 需要清楚定義 canonical source，否則容易出現版本差異 |

建議優先評估：`Markdown 為權威月報格式，PDF 作為第二階段派生輸出`。

理由：§11 已定 Markdown 為最低要求，且 repo 目前全為 Markdown 文件。先穩定 Markdown 模板，能確保四問摘要、觸發訊號、情境更新與資料品質不漏項。PDF 可等模板與審核流程穩定後再由 Markdown 生成，不應手工維護另一份內容。

決議前需確認：

- 月報主要是自用、內部分享，還是需要正式對外文件？
- 是否需要圖表與表格在 PDF 中保持固定版面？
- 是否接受「Markdown 為唯一可編輯源，PDF/HTML 只做衍生檔」？
- 每月報告是否需要存放在固定資料夾與固定命名規則？

## 6. 情境機率

待決問題：是否保留來源報告的 30/45/25 起始機率，還是只顯示 active/watch/invalidated？

判斷準則：

- §7 已明確規定情境由條件計數與否證條件判斷，不是用單一分數決定。
- 30/45/25 若未經月度資料校正，容易被誤讀為精準機率。
- 本儀表板的內容聲調要求保留不確定性，避免投資建議或精準預測語氣。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| 保留 30/45/25 為主畫面機率 | 需要延續來源報告原始敘事 | 使用者能快速理解初始 Bear/Base/Bull 權重 | 容易被誤讀為模型輸出或即時預測；若無回測校正，精度不足 |
| 只顯示 active/watch/invalidated | 重視可稽核條件與避免偽精準 | 與 §7 情境引擎最一致；降低誤讀 | 對需要配置語言的使用者，可能覺得資訊不足 |
| 保留為「研究先驗」附註 | 想保留來源報告脈絡，但不當作本月輸出 | 同時保存原始敘事與儀表板紀律；可放在 tooltip/附錄 | 需要文案明確區分先驗、狀態與模型機率 |
| 改成低/中/高信心區間 | 希望提供方向感但避免精確百分比 | 比百分比更保守；可搭配證據完整度 | 若規則未定義清楚，仍可能變成主觀標籤 |

建議優先評估：`主畫面只顯示 active/watch/invalidated；30/45/25 僅作為研究先驗附註`。

理由：目前 §7 的情境引擎是條件式判斷，尚未在 Phase 3 以歷史樣本校正機率。把 30/45/25 放在主畫面，會讓使用者以為每月資料已產生可量化機率。較穩健的做法是主畫面只顯示 Bear/Base/Bull 狀態與證據，並在附註中說明來源報告的起始情境權重。

決議前需確認：

- 使用者是否真的需要百分比語言來做資產配置討論？
- 若保留 30/45/25，是否要標註「來源報告起始假設，非本月模型輸出」？
- Phase 3 回測前，是否禁止自動調整情境機率？
- 是否需要另設 `scenario_prior` 與 `scenario_state` 兩種欄位，避免混用？

## 7. 更新責任

待決問題：資料輸入、研究判讀、月報審核是否由同一人完成？

判斷準則：

- 資料輸入錯誤、質性判讀偏誤、月報文案過度肯定，是三種不同風險。
- 即使單人操作，也應在流程上拆成資料、判讀、審核三個步驟。
- 若月報會被分享或用於決策會議，至少應有第二人或第二輪審核。

| 選項 | 適合情境 | 優點 | 主要風險 |
|---|---|---|---|
| 同一人完成全部流程 | 個人研究、低頻自用 | 成本最低；迭代最快；責任清楚 | 容易把輸入錯誤與判讀偏誤帶進月報；缺少反證檢查 |
| 資料輸入、研究判讀、月報審核分工 | 內部團隊、正式月報 | 可降低錯誤；符合可稽核要求；責任清楚 | 需要協調流程與時間；可能拖慢發布 |
| 單人操作但角色拆分 | 個人或小團隊過渡期 | 不需要多人，也能用 checklist 強制分步；易落地 | 審核仍非獨立；對抗式檢查效果有限 |
| 季度第二方審查 | 結論轉向或季度末 | 能檢查否證門檻是否被忽略；符合 SOP §4 的對抗式審查精神 | 不適合每月高頻依賴；需要找得到有能力的審查者 |

建議優先評估：`角色拆分的輕量 RACI：Data Owner、Research Owner、Reviewer`。

具體方向：資料輸入者負責 source、data_as_of、freshness 與欄位完整性；研究判讀者負責 signal_state、analyst_note 與情境狀態；審核者負責確認月報是否漏列 stale/missing、是否過度投資建議化、是否正確揭露暫估與覆蓋率。若短期只有一人，也應在月度 checklist 中分三步簽核，避免同一時間完成全部判斷。

決議前需確認：

- 月報是否會被他人閱讀或用於會議？
- 是否存在可擔任第二審核者的人？
- FINRA 發布後的正式重算是否需要重新審核？
- 月報是否需要記錄 `prepared_by`、`reviewed_by`、`approved_at`？

## 8. 建議彙總表

| 待決問題 | 建議優先評估方向 | 是否為最終決議 | 主要依賴 |
|---|---|---|---|
| 實作載體 | Streamlit 本地 MVP + Markdown/CSV/JSON 權威層 | 否 | 使用者型態、部署需求、是否接受本機 Python |
| 資料源授權 | 混合模式；Phase 1 可在公開資料 + 手動 CSV 下運作 | 否 | 是否有資料商授權與再利用權限 |
| 資料抓取模式 | 混合抓取；Phase 1 以 manual_csv 作為 canonical input | 否 | 資料源分類、欄位驗證、錯誤處理 |
| 腳本執行載體 | Phase 1 手動執行留 log；穩定後再選 GitHub Actions 或本機 cron | 否 | 資料是否公開、是否含本機或付費來源 |
| 月報匯出格式 | Markdown 為權威格式，PDF 作為派生輸出 | 否 | 分享需求、版面要求、報告保存規則 |
| 情境機率 | 主畫面只顯示 active/watch/invalidated；30/45/25 僅作研究先驗附註 | 否 | 是否需要百分比語言、Phase 3 校正計畫 |
| 更新責任 | 輕量 RACI：Data Owner、Research Owner、Reviewer | 否 | 是否多人協作、是否需要正式審核 |

## 9. 不建議現在做的事

- 不建議直接把任一建議寫回「已決議」，因為使用者尚未拍板。
- 不建議在資料授權未確認前選定全自動雲端排程。
- 不建議在 Phase 3 回測前，把 30/45/25 包裝成本月模型機率。
- 不建議在 §14 尚未清空前開始 Phase 1 實作或把 `status: draft` 改成 `status: spec`。
