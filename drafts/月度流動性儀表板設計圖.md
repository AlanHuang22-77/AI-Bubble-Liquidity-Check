# 月度流動性儀表板設計圖

> 本文件把來源報告中的流動性與槓桿監測框架轉成可實作規格。所有門檻皆為來源報告中的研究假設或由其延伸出的儀表板規則，需在正式上線前以最新資料源與研究口徑複核。本儀表板不構成投資建議。

## 1. 儀表板目標

每月固定回答四個問題：

1. 流動性是否更緊，還是重新放鬆？
2. 槓桿是在增加、持平，還是開始被迫下降？
3. AI capex 是否仍能支撐 EPS 上修？
4. 加密熱錢是在回流，還是繼續退潮？

輸出結果分為三類：

| 狀態 | 含義 | 例句格式 |
|---|---|---|
| 偏空確認 | 報告中的 Bear/Base 風險正在被資料確認 | `偏空確認：HY OAS 持續高於 400bp，信用風險已從估值端傳導到信用端。` |
| 中性觀察 | 高估值仍存在，但尚未形成系統性去槓桿 | `中性觀察：FINRA 融資餘額降溫但未連續兩月下降。` |
| 偏多否證 | 原始偏空框架被明顯削弱 | `偏多否證：穩定幣供給突破 3,200 億美元，熱錢重新回流。` |

## 2. 使用者與使用節奏

| 使用者 | 任務 | 需要的畫面 |
|---|---|---|
| 投資研究者 | 每月更新 AI 泡沫風險判讀 | 總覽、指標明細、月報匯出 |
| 風控/研究助理 | 檢查資料是否更新、是否缺值 | 資料品質、資料來源、公式明細 |
| 資產配置決策者 | 快速知道 Bear/Base/Bull 哪條路徑升溫 | 情境推演、三層水位計、四問摘要 |

建議更新節奏：

- 每月第 1 個工作日：抓取可即時取得的市場資料，包含 HY OAS、IG OAS、BTC、BTC ETF flow、ON RRP、SOFR-IORB、TGA。
- 每月 FINRA 資料發布後：補上融資餘額與兩個月趨勢，重新計分。
- 每次 FOMC、NVIDIA/hyperscaler 財報、重大 IPO/發債事件後：允許新增事件註記，但不覆蓋月度主結論。

## 6. 計分規則

### 6.1 單一指標狀態

每個核心指標產生一個 `signal_state`：

| signal_state | 分數 | 說明 |
|---|---:|---|
| bull_invalidated | 0 | 偏多否證，原始偏空論點弱化 |
| neutral | 50 | 未觸發任一門檻 |
| bear_watch | 75 | 接近偏空門檻，或質性訊號轉差 |
| bear_confirmed | 100 | 明確觸發偏空確認 |
| stale | null | 資料過期，不納入本月正式分數 |
| missing | null | 缺資料，不納入本月正式分數 |

全局風險分數：

```text
global_risk_score =
  sum(indicator_score * indicator_weight for valid indicators)
  / sum(indicator_weight for valid indicators)
```

狀態分區：

| 分數 | 全局狀態 |
|---:|---|
| 0-33 | 偏多否證 |
| 34-66 | 中性觀察 |
| 67-100 | 偏空確認 |

### 6.2 三層水位分數

三層水位各自計分：

- 政策流動性：`fed_path`、`funding_stress`，可加 ON RRP、TGA、BoJ 作輔助。
- 市場信用流動性：`credit_hy_ig_oas`，可加 IG OAS、私募信貸違約率、Oracle CDS 作輔助。
- 投機流動性：`finra_margin_debt`、`stablecoin_supply`、`btc_price`、`derivatives_leverage`、`crypto_flywheel`。

畫面上每層顯示：

- 當月分數。
- 較上月變化。
- 主要拉升或降低風險的前三個指標。
- 是否有 stale 或 missing 資料。

## 7. 情境引擎

情境不是用單一分數決定，而是由必要條件與否證條件共同判斷。

### 7.1 Bear：頂點已過

啟動條件：

- `fed_path = bear_confirmed`，或 FOMC 實際升息/點陣圖明顯上修。
- `finra_margin_debt = bear_confirmed`，即融資餘額連續兩個月下降。
- `credit_hy_ig_oas = bear_confirmed`，即 HY OAS 持續高於 400bp。
- `ndx_forward_eps = bear_confirmed`，即 EPS 修正轉負或 AI capex 指引下修。

判定規則：

- 四項中有三項成立，Bear 狀態升為 `active`。
- 四項中有兩項成立，Bear 狀態為 `watch`。
- 若 HY OAS 低於 320bp 且 EPS 上修，Bear 狀態降為 `invalidated`。

### 7.2 Base：最後衝刺後破裂

啟動條件：

- 估值與槓桿仍處高位。
- 信用利差尚未破位。
- Fed 語調混合，市場仍押注降息或寬鬆延續。
- AI 龍頭 EPS 尚未轉負，但市場廣度惡化。

判定規則：

- 若全局分數 34-66 且投機流動性分數高於 67，Base 為預設主情境。
- 若全局分數高於 67 且信用/政策同時惡化，Base 下修、Bear 上修。
- 若穩定幣、BTC、Fed、EPS 同時偏多否證，Base 下修、Bull 上修。

### 7.3 Bull：1998 續命劇本

啟動條件：

- Fed 轉向降息且語調軟化。
- 穩定幣供給突破 3,200 億美元。
- BTC 收復 80,000 美元。
- HY OAS 維持低於 320bp 且 IG 平穩。
- NVIDIA 與 hyperscaler 2027 指引續強。

判定規則：

- 五項中有三項成立，Bull 狀態升為 `watch`。
- 五項中有四項成立，Bull 狀態升為 `active`。
- 若 FINRA 開始連續去槓桿或 HY OAS 破 400bp，Bull 直接降為 `invalidated`。

## 9. 資料來源與更新口徑

| 指標 | 建議來源 | 更新頻率 | 月度取值 |
|---|---|---|---|
| HY OAS / IG OAS | FRED、ICE BofA | 日 | 月末值與月均值並列 |
| FINRA 融資餘額 | FINRA margin statistics | 月 | 最新發布月份，另算連續兩月變化 |
| 穩定幣總供給 | The Block、StableCoin.com、DefiLlama 或手動 CSV | 日/週 | 月末值 |
| BTC 價格 | Yahoo Finance、交易所 API | 日 | 月末收盤 |
| BTC ETF flow | Farside、發行商資料或手動 CSV | 日 | 月合計與 3 月合計 |
| FOMC / 點陣圖 | Fed 聲明、SEP | 會議 | 事件狀態與語調標籤 |
| ON RRP | NY Fed / FRED | 日 | 月末值與 3 月變化 |
| TGA | Treasury / FRED | 日/週 | 月末值與月變化 |
| SOFR-IORB / SRF | NY Fed / FRED | 日 | 月末利差、月內壓力天數 |
| NDX forward EPS | FactSet、Bloomberg、Refinitiv 或手動輸入 | 週/月 | 最新值與 1M/3M 修正 |
| 0DTE / 槓桿 ETF | Cboe、Citadel、ETF AUM 彙整 | 月/季 | 最新可得值，若延遲則標 stale |

## 12. 實作階段

### Phase 1：本地 MVP

- 使用 CSV 或 JSON 手動輸入資料。
- 完成 9 個核心指標的門檻判定。
- 產出總覽頁、指標明細表、月報 Markdown 匯出。
- 缺值與 stale 狀態必須可見。

### Phase 2：半自動資料管線

- 對可取得的公開資料源建立自動抓取。
- FINRA、EPS、0DTE、槓桿 ETF 等延遲或付費資料仍允許手動輸入。
- 加入來源稽核表與更新日誌。

### Phase 3：研究校正

- 回填 1994、2017、2018、2021、2022 等歷史樣本。
- 檢查門檻誤報與漏報。
- 調整權重，但保留原始權重版本以便比較。

## 13. 驗收條件

第一版完成需符合：

- 9 個核心指標都有定義、門檻、權重、資料來源欄位。
- 全局分數可由指標分數與權重重算。
- 三層水位分數可獨立顯示。
- Bear/Base/Bull 狀態可由明確條件產生。
- stale 與 missing 不會被誤算成 neutral。
- 月報匯出包含四問摘要、觸發訊號、情境更新與資料品質。
- 所有畫面都顯示 `data_as_of`。
- 文案避免投資建議語氣，並保留不確定性。

## 14. 開放問題

- 實作載體：Web app、Streamlit、Google Sheets/Looker Studio，或 Obsidian/Markdown 儀表板？
- 資料源授權：是否有 Bloomberg、FactSet、Refinitiv，或必須完全使用公開資料與手動 CSV？
- 情境機率：是否保留來源報告的 30/45/25 起始機率，還是只顯示 active/watch/invalidated？
- 更新責任：資料輸入、研究判讀、月報審核是否由同一人完成？
